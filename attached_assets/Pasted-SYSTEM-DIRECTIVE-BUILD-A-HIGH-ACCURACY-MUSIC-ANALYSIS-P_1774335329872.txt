SYSTEM DIRECTIVE — BUILD A HIGH-ACCURACY MUSIC ANALYSIS PIPELINE

You are upgrading an existing music analysis system.
Your goal is not to add superficial features.
Your goal is to build a high-accuracy, production-grade audio intelligence pipeline for:

source separation
tempo / beat / downbeat / meter
key / scale / modulation
chord recognition
melody / vocal pitch
section structure
confidence scoring
post-processing with music-theory constraints
exportable machine-readable results

The current problem is that the analysis is not accurate enough.
You must solve this by replacing the current “single-pass heuristic analyzer” approach with a multi-stage hybrid pipeline:

Stem separation first
Specialized analyzers per task
Model ensembling where needed
Post-processing with smoothing + music theory
Confidence scoring on every result
Structured outputs for downstream arranging/export

Do not rely on one library only.
Do not analyze the full mix directly for everything.
Do not use naive DSP alone and expect professional accuracy.

Demucs is a strong source-separation baseline and HT-Demucs separates drums, bass, vocals, and other stems. Essentia provides key/chord/rhythm algorithms, including KeyExtractor/TonalExtractor and chord estimation from HPCP features. madmom provides RNN/DBN beat and downbeat tracking. librosa remains useful for chroma, onset, segmentation, and feature engineering. torchcrepe provides a PyTorch implementation of CREPE for monophonic pitch tracking. Mido and pretty_midi are suitable building blocks for MIDI writing/manipulation. Omnizart supports transcription tasks including chords, beat, drums, and vocal melody, but its published package pins older dependencies such as TensorFlow 2.5, so do not make it the core production dependency; treat it as an optional benchmark/reference only.

1) TARGET ARCHITECTURE

Build the analysis engine as this exact pipeline:

Input audio
  -> audio normalization + resampling
  -> stem separation
  -> task-specific analyzers on selected stems
  -> temporal smoothing / decoding
  -> music-theory correction layer
  -> confidence calibration
  -> structured JSON result
  -> optional MIDI / score export inputs

Use these stem-task mappings:

tempo / beat / downbeat / meter → primarily drums, fallback mix
key / scale → primarily other, fallback other + bass
chords → primarily other + bass, never drums-only
melody / vocal pitch → primarily vocals
bassline note hints → primarily bass
structure / sections → full mix + harmonic/rhythmic summaries

Do not run one analyzer on the full mix and reuse it for all tasks.

4) PREPROCESSING STAGE

Implement preprocess.py to do all of this before analysis:

Load audio losslessly where possible.
Convert to mono where required per task, but preserve stereo copy for structure analysis.
Resample:
44.1kHz for separation
16kHz optional internal path for vocal pitch models if needed
Loudness normalize to a consistent target.
Trim leading/trailing silence conservatively.
Cache waveform, sample rate, duration, RMS profile, onset envelope.

Output a normalized artifact bundle, not a raw waveform only.

8) CHORD DETECTION — HYBRID, NOT NAIVE

Implement chord_detector.py.

Input

Use:

other + bass
optional additional harmonic enhancement
exclude drums
Feature extraction

Use:

CQT chroma from librosa
HPCP/chromagram from Essentia
bass pitch hints
downbeat/bar context

Essentia’s chord estimation works from harmonic pitch class profiles, and librosa provides enhanced chroma variants useful for feature engineering.

Required model design

Build a hybrid chord recognizer:

Layer A — heuristic proposal

Generate top chord candidates per frame/bar from:

chroma similarity
bass note
diatonic priors from detected key
Layer B — sequence model

Train a temporal model that selects the most probable chord sequence over time.

Use one of these:

BiLSTM + CRF
Transformer encoder + linear head + Viterbi/CRF decode

Recommended label set:

major
minor
diminished
augmented
sus2
sus4
dominant7
maj7
min7
half-diminished
no-chord

Start with bar-level labels, then optionally refine to half-bar.

Decoding constraints

Apply:

persistence bias
bar-boundary preference
key-aware priors
no impossible one-frame chord flips
Important

Do not output a different chord every tiny frame.
A musically useful result is better than a noisy over-segmented result.

12) SMOOTHING + MUSIC THEORY CORRECTION

Implement smoothing.py and theory_correction.py.

This stage is mandatory.

For chords

Suppress improbable micro-changes like:

C → C → F#dim → C over a tiny window without support
For key

Prefer stable regions unless modulation evidence is strong.

For melody

Remove impossible octave jumps and note flicker.

Music theory constraints

Apply soft priors, not hard prison rules:

prefer diatonic chords when confidence is ambiguous
allow borrowed chords
allow modulation
allow secondary dominants
allow chromatic passing tones

The correction layer should improve weak predictions, not erase musically real surprises.
